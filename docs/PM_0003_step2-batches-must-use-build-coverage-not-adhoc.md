# PM-0003: Ad-Hoc Batch Assembly Bypassed Pipeline, Producing Empty Context

## What Happened

Step 2 evaluation of ~676 ch3 findings was run using ad-hoc batch files assembled inline by agents, bypassing `build_coverage.py --embed-prep`. The ad-hoc batches used wrong key names and contained empty values for all critical context fields:

| Field | Script output (correct) | Ad-hoc output (broken) |
|-------|------------------------|----------------------|
| Q&A history | `qa_history`: 14,407 chars | `qa_content`: 0 chars |
| Anti-pattern rules | `anti_pattern_rules`: populated | `anti_patterns`: 0 chars |
| Chapter inventory | `chapter_inventory`: 53 arguments | `inventory`: empty `[]` |
| Cross-chapter inventories | `cross_chapter_inventories`: all chapters | missing entirely |

All 13 ad-hoc batches (batch2–batch18) were affected. Only `pilot` and `top80` (generated by the script) had correct context.

## Impact

- 676 findings evaluated blind — no Q&A decision history, no anti-pattern rules, no argument-level inventory, no cross-chapter routing
- ~30% verdict variance across runs on identical data
- 29 findings confirmed to have flipped between RESEARCH and SKIP_restates across runs
- Multiple valuable primary source findings (Philo's "deuteros theos", Priene inscription, Josephus christos for plastering, kyrie eleison) wrongly classified as SKIP_restates
- All ad-hoc Step 2 verdicts must be discarded and re-evaluated

## Root Cause

`build_coverage.py --embed-prep` exists and correctly assembles context packages. But nothing prevented agents from constructing batch dicts inline with arbitrary key names. The agent assembled JSON objects directly in its prompt, guessing at field names, and never validated that the fields contained data.

The deeper cause: the pipeline has a build step (the script) but no validation step. Any JSON object that looks like a batch gets evaluated, even if it's missing every piece of context the evaluator needs.

## Action Items

### [mitigate-this-incident]
1. Discard all ad-hoc Step 2 verdicts (batch2–batch18, cleanup)
2. Regenerate all Step 2 batches using `build_coverage.py --embed-prep`
3. Re-evaluate all findings with proper context

### [prevent]
4. Add a batch validation function to `build_coverage.py` that checks:
   - Required keys exist: `qa_history`, `anti_pattern_rules`, `chapter_inventory`, `cross_chapter_inventories`, `chapter_text`, `step2_rules`, `findings`
   - `qa_history` is non-empty (every chapter has at least global Q&A)
   - `anti_pattern_rules` is non-empty
   - `chapter_inventory` has at least 1 entry
   - `chapter_text` has at least 100 chars
   - `findings` is a non-empty list
5. Add a `--validate` command to `build_coverage.py` that validates all batch files in a directory. Run before any evaluation.
6. Add to CLAUDE.md / pipeline instructions: **Step 2 batches MUST be generated by `build_coverage.py --embed-prep`. Ad-hoc batch assembly is forbidden. Any batch file not passing `--validate` must not be evaluated.**

### [detect]
7. The validate function should be called automatically at the start of any evaluation agent prompt, before processing findings. If validation fails, the agent must stop and report the failure.
