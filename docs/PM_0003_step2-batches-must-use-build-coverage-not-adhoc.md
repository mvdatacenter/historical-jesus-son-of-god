# PM-0003: Ad-Hoc Batch Assembly Bypassed Pipeline

## What Happened

Step 2 evaluation of ~676 ch3 findings was run using ad-hoc batch files assembled inline by agents, bypassing the private research repo's `build_coverage.py --embed-prep`. The ad-hoc batches used wrong key names and contained empty values for all required context fields:

| Field | Script output (correct) | Ad-hoc output (broken) |
|-------|------------------------|----------------------|
| Q&A history | `qa_history`: 14,407 chars | `qa_content`: 0 chars |
| Anti-pattern rules | `anti_pattern_rules`: populated | `anti_patterns`: 0 chars |
| Chapter inventory | `chapter_inventory`: 53 arguments | `inventory`: empty `[]` |
| Cross-chapter inventories | `cross_chapter_inventories`: all chapters | missing entirely |

All 13 ad-hoc batches (batch2–batch18) were affected. Only `pilot` and `top80` (generated by the script) had correct context.

This left 676 findings missing Q&A decision history, anti-pattern rules, argument-level inventory, and cross-chapter routing.
Across repeated runs, verdicts varied by ~30%, 29 findings flipped between `RESEARCH` and `SKIP_restates`, and primary-source findings such as Philo's *deuteros theos*, the Priene inscription, Josephus's *christos*, and *kyrie eleison* were wrongly classified as `SKIP_restates`.
All ad-hoc Step 2 verdicts had to be discarded and re-evaluated.

## Root Cause

The private research repo's `build_coverage.py --embed-prep` exists and assembles context packages correctly, and the pipeline still allowed agents to construct batch dicts inline with arbitrary key names. The agent assembled JSON objects directly in its prompt, guessed at field names, and skipped validation that the fields contained data.

The pipeline had a build step and lacked a validation step. Any JSON object that looked like a batch was evaluated, even when it was missing every piece of context the evaluator needed.
